# Правило: обязательная маршрутизация запроса к агенту

Для **каждого нового пользовательского запроса** сначала определи, какому агенту из `.cursor/agents` он соответствует, и только потом отвечай.

## Обязательный протокол

1. Определи основной intent запроса (requirements, архитектура, backend, frontend, QA, DevOps, SDK, UX, docs, open-source).
2. Выбери **ровно одного** primary-агента:
   - `po` — требования, приоритизация, scope, backlog
   - `architect` — архитектурные решения, ADR, границы модулей
   - `dev` — общая реализация и багфиксы
   - `nest-engineer` — backend/Nest.js/API/сервисы
   - `react-next-engineer` — frontend/React/Next.js/UI
   - `qa` — тест-кейсы, регресс, валидация качества
   - `devops` — CI/CD, инфраструктура, окружение, скрипты
   - `sdk` — SDK, контракты, интеграции
   - `ux` — UX-потоки, интеракции, usability
   - `tech-writer` — документация, runbook, release notes
   - `open-source` — OSS-подготовка, community, contribution
3. Считай, что выбранный агент "загружен в чат": работай по его роли и ограничениям из `.cursor/agents/<agent>.md`.
4. Если запрос смешанный:
   - выбери агента по **доминирующей** части задачи;
   - при необходимости укажи, каких вторичных агентов нужно подключить следующим шагом.
5. Если intent неясен — выбери `orchestrator`, уточни минимально необходимое и зафиксируй план маршрутизации.

## Формат начала ответа

В начале каждого ответа добавляй строку:

`[Agent Router] primary=<agent-id>`

Пример:

`[Agent Router] primary=react-next-engineer`

## Ограничения

- Нельзя пропускать этап маршрутизации.
- Нельзя одновременно объявлять несколько primary-агентов.
- Если выбран `orchestrator`, он координирует и делегирует, а не заменяет профильных агентов.
